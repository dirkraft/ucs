## Project: UCS (Universal Container Session)

Dispatcher that maps Slack threads ‚Üí Docker containers ‚Üí persistent Claude Code sessions.
Stack: Python (dispatcher), Vue/Vite (UI, future).

## Design Decisions

### 1. Session continuity ‚Äî `--resume root`

Each container represents a single ongoing session. Two-phase invocation pattern (verified working on CC 2.1.107):

- **Container didn't exist** (just created it): `claude --print --verbose --name root <prompt>`
- **Container already existed**: `claude --print --verbose --resume root <prompt>`

Container existence is the state ‚Äî no extra tracking needed.

### 2. Interruption ‚Äî SIGTERM to the `claude` process

`docker exec claude ...` produces a child process inside the container. On new message arrival while agent is active: send SIGTERM to that process. In interactive CC, Esc/Ctrl-C gracefully interrupt active thinking without corrupting state ‚Äî expectation is SIGTERM achieves the same. Track the `docker exec` PID on the host side so we can signal it. Verify behavior empirically once we have a running container.

### 3. Container lifecycle ‚Äî idle timeout

- Agent invocations always go through `docker exec` into a running container (never `docker run` per invocation).
- This keeps dispatcher code clean and allows manual `docker exec` shells into live containers.
- After no active agent process for **5 minutes**, stop the container.
- Implementation options (open): poll for active `claude` process inside container on a timer; or track last-active timestamp in dispatcher and use `asyncio` / a background task to stop after idle threshold.
- Container is *stopped* not *removed* ‚Äî `docker start` brings it back on next message.

### 4. System prompt ‚Äî deferred

`--append-system-prompt` is more appropriate than `--system-prompt` (doesn't override defaults). Skip for now. Will matter when tackling richer reasoning: channel search, Slack API retrieval, etc. Revisit once core loop is working.

### 5. Stream parsing

ndjson from `claude --output-format stream-json --print --verbose`:

| event type | action |
|---|---|
| `assistant` / `thinking` content | append `* <text>` bullet to Slack placeholder |
| `assistant` / `tool_use` content | append `* calling <name>...` bullet |
| `assistant` / `text` content | buffer |
| `result` (success) | replace entire placeholder with `result.result` |
| `result` (error) | post error to thread |

The `result.result` field always contains the clean final response ‚Äî use that, not the accumulated text buffer.

## Docker Setup

- Image name: `ucs_agent_claude` (Debian base + Claude Code CLI)
- Container naming: `ucs_sess_slack_<thread_id>`
- Memory limit: 4GB (`--memory 4g`)
- Credentials: copy `~/.claude/.credentials.json` into container at start
- Always interact via `docker exec`, never `docker run` per-prompt

## Slack Bot

Credentials available via `.envrc` ‚Üí `.env`.
Dispatcher flow on `@pal` mention:
1. Verify authorized user ‚Äî ignore if not
2. Create/start container if needed
3. Interrupt active agent if running (SIGTERM) ‚Üí post "interrupting..." to placeholder
4. Update placeholder to "thinking..."
5. `docker exec claude --dangerously-skip-permissions --output-format stream-json --print --verbose --resume root <prompt>`
6. Parse stream ‚Üí live-edit placeholder with reasoning bullets
7. On `result` ‚Üí replace placeholder with final response
